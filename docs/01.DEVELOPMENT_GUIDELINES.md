## Branching Strategy

We adopt a **phased branching approach** that evolves with the product maturity — starting lean with Trunk-Based Development, then graduating to a Simplified Git Flow once the product reaches production scale.

---

### Overview

| Phase | Strategy | When |
|---|---|---|
| **Phase 1** | Trunk-Based Development | Startup / pre-production |
| **Phase 2** | Simplified Git Flow | Cycle3/Cycle4 or immediately after production launch |

---

### Ticket ID

> 📋 **`[Ticket-ID]`** refers to the issue identifier from our project tracker on **Plane**.
>
> 🔗 [CleanTech Thailand — Issues Board](https://app.plane.so/cleantech_thailand/projects/b85ed333-2118-4b0b-9948-48e17fce22ba/issues/)

All branch names must include the Ticket ID from Plane so that every branch is traceable back to a specific issue.

---

### Phase 1 — Trunk-Based Development (Startup)

> **Goal:** Move fast, stay lean, reduce merge overhead.

During the early startup phase, all engineers work close to `main`. Branches are **short-lived**, small in scope, and merged frequently to avoid long-running divergence.

#### Branch Naming

```
feature/[Ticket-ID]
```

**Examples:**
```
feature/CCT-1
feature/CCT-2
```

#### Rules

- ✅ Branch off from `main`
- ✅ Keep branches **small and focused** — ideally < 1–2 days of work
- ✅ Open a **Pull Request (PR)** for every merge to `main`
- ✅ Require **at least 1 approval** before merging
- ✅ Delete branch after merge
- ❌ Do not let branches live longer than **3 days** without merging or rebasing

#### Flow Diagram

```
main ──────●──────────●──────────●──────────▶
            ↑          ↑          ↑
       feature/       feature/   feature/
       CCT-1          CCT-2      CCT-3
```

### Phase 2 — Simplified Git Flow (C3/C4 / Post-Launch)

> **Goal:** Protect production stability while maintaining development velocity.

Once the product is live (or approaching C3/C4 milestone), we introduce a structured branching model with clear separation between development, integration, and production environments.

#### Branch Structure

| Branch | Purpose | Protection |
|---|---|---|
| `main` | Production-ready code. **Only** released from `develop`. | ✅ Protected — no direct push |
| `develop` | Integration branch. All features merge here first. | ✅ Auto-merge after PR approval |
| `feature/[Ticket-ID]` | Individual feature development | — |
| `hotfix/[Ticket-ID]` | Urgent production bug fixes | — |

#### Branch Naming

```
feature/[Ticket-ID]     # New features
hotfix/[Ticket-ID]      # Production hotfixes
```

**Examples:**
```
feature/CT-9
hotfix/CT-10
```

#### Rules

- ✅ All **features** branch from `develop` and merge back to `develop` via PR
- ✅ `main` is **only updated** by merging from `develop` (release) or a `hotfix/*` branch (emergency)
- ✅ **Hotfixes** branch from `main`, and must be **back-merged to `develop`** after release
- ✅ PRs to `develop` require **at least 1 approval** + passing CI
- ✅ PRs to `main` require **at least 2 approvals** + passing CI
- ❌ No direct commits to `main` or `develop`

#### Flow Diagram

```
main     ──────●────────────────────────────────●──────────▶
                ↑  (release)                     ↑ (hotfix)
develop  ──────●──────●──────●──────────────────●──────────▶
                       ↑      ↑                  ↑
                  feature/  feature/         hotfix/
                  CT-8      CT-9             CT-10
```

#### Hotfix Flow

```
main ──────●─────────────────────●──────────▶
            \                   ↑↑
             hotfix/CT-10 ────┘ \
                                  ↓
develop ───────────────────────────●──────────▶
                           (back-merge hotfix)
```

### Quick Reference

```bash
# Phase 1 — Create a feature branch (Trunk-Based)
git checkout main
git pull origin main
git checkout -b feature/CT-5

# Phase 2 — Create a feature branch (Git Flow)
git checkout develop
git pull origin develop
git checkout -b feature/CT-6

# Phase 2 — Create a hotfix branch
git checkout main
git pull origin main
git checkout -b hotfix/CT-10
```

---

> 💡 **When in doubt, keep branches small.** The best PR is one that's easy to review, easy to understand, and easy to revert.
