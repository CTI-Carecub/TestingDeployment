## Review SLA

Timely reviews keep branches short-lived (our core Trunk-Based Development principle) and prevent merge debt from accumulating.

| PR Type | First Response | Full Review |
|---|---|---|
| **Feature / Chore** | ≤ 4 business hours | ≤ 1 business day |
| **Hotfix** | ≤ 1 hour | ≤ 2 hours |

### Rules

- **Authors** must request review immediately after opening a PR — do not leave it in draft longer than needed.
- **Reviewers** must acknowledge the review request within **4 business hours**.
- If a reviewer cannot complete the review within the SLA, they must **re-assign** or **notify the team** in the project channel.
- PRs that have been approved and pass CI must be **merged within 1 business day** by the author. Do not leave approved PRs sitting idle.
- A PR open for more than **3 days** without activity must be escalated to the team lead.

## Blocking vs. Non-Blocking Comments

All review comments must be prefixed with a label so the author knows exactly what action is required. This eliminates ambiguity.

### Prefix Convention

| Prefix | Meaning | Merge-blocking? |
|---|---|---|
| `[BLOCKING]` | Must be resolved before approval is granted | ✅ Yes |
| `[SUGGESTION]` | Improvement idea — author decides whether to act | ❌ No |
| `[QUESTION]` | Seeking understanding — no change required unless the answer reveals a problem | ❌ No |
| `[FYI]` | Informational note, no action needed | ❌ No |


### Resolving Comments

- **Authors** resolve comments after addressing them — not reviewers.
- A resolved thread means the author has acted on it (code changed, or a decision was made to not change it with a written explanation).
- Do not resolve threads by just replying "done" without making the corresponding change visible.
- **Blocking comments take priority** — do not push the reviewer to re-approve until all `[BLOCKING]` threads are resolved.

## Approval Criteria

Approval requirements follow the branching phase defined in [01.DEVELOPMENT_GUIDELINES.md](./01.DEVELOPMENT_GUIDELINES.md).

### Phase 1 — Trunk-Based Development

| Target Branch | Required Approvals | CI Required |
|---|---|---|
| `main` | ≥ 1 approval | ✅ All 3 CI jobs must pass |

### Phase 2 — Simplified Git Flow (C3/C4 / Post-Launch)

| Target Branch | Required Approvals | CI Required |
|---|---|---|
| `develop` | ≥ 1 approval | ✅ All 3 CI jobs must pass |
| `main` | ≥ 2 approvals | ✅ All 3 CI jobs must pass |

### CI Gate (Required for All PRs)

All three CI jobs defined in `.github/workflows/ci.yml` must pass before merging:

| Job | Command | What it checks |
|---|---|---|
| **Lint & Format** | `pnpm run ci` | Biome lint + format rules |
| **Unit Test & Coverage** | `pnpm run test:coverage` | Vitest tests + coverage thresholds |
| **Build** | `pnpm run build` | Next.js production build |

> A passing CI is **not** a substitute for a human review. Approving a PR only because CI is green is not acceptable.

### What "Approved" Means

An approval signals that the reviewer:

1. Understands what the change does and why.
2. Has no remaining `[BLOCKING]` comments.
3. Is confident the change is safe to merge to the target branch.
4. Has verified the PR against the [Reviewer Checklist](#reviewer-checklist).

An approval **does not** mean the reviewer has read every line in microscopic detail — it means the reviewer is satisfied with the overall quality, correctness, and safety of the change.

### Re-review After Changes

- If an author pushes significant changes after a review cycle, they must **re-request review** explicitly.
- A previous approval is **invalidated** if new commits touch core logic, security-sensitive code, or data access patterns.
- Minor fixes (typos, comment clarifications) do not require a new review cycle — use your judgment.

## Code Review Guidelines

### What to Look For

Focus your review energy in this order:

#### 1. Correctness & Logic
- Does the code do what the PR description claims?
- Are there edge cases, off-by-one errors, or null/undefined paths that could cause runtime errors?
- Are async operations handled correctly (loading, error, and success states)?

#### 2. Security
- No secrets, API keys, tokens, or credentials committed.
- No sensitive data exposed in client bundles (check `'use client'` boundaries).
- Supabase Row Level Security (RLS) policies considered for any query or schema changes.
- No direct user input passed to queries or shell commands without sanitisation (SQL injection, XSS, command injection).

#### 3. Next.js App Router Conventions
- New components are **Server Components by default** — `'use client'` is added only where interactivity or browser-only APIs (hooks, event handlers) are required.
- Data fetching happens in Server Components or Server Actions — not in `useEffect` unless justified.
- Route handlers follow `app/api/**/route.ts` conventions.
- No heavy client-side libraries included in the server bundle unnecessarily.

#### 4. Supabase & Data Access
- Queries are protected by RLS or explicit auth checks.
- No overly broad `select *` queries where specific columns are sufficient.
- Error responses from Supabase are handled (check for `.error` on returned results).

#### 5. TypeScript
- No use of `any` unless there is a documented justification.
- Types are explicit and meaningful — avoid overly generic types.
- Exported types/interfaces are consistent with the rest of the codebase.

#### 6. Testing
- Unit tests are added for new logic.
- Tests cover happy path, edge cases, and error cases where relevant.
- Tests are readable and their intent is clear.

#### 7. Code Quality & Maintainability
- Code is simple and focused — no over-engineering or premature abstractions.
- No dead code, commented-out blocks, or `console.log` / debug artifacts.
- Naming is clear and consistent with project conventions.
- Changes are scoped to the stated purpose — no unrelated refactors bundled in.

### What NOT to Block On

Do not block a PR over subjective style preferences that Biome already enforces automatically. If Biome passes, formatting and most style rules are satisfied. Reserve `[BLOCKING]` for correctness, security, and significant design concerns.

### Tone & Collaboration

- Review the **code**, not the person.
- Be specific — point to lines, not vague areas ("this function looks wrong" is unhelpful; "line 42 — `userId` could be `undefined` here if the session hasn't loaded yet" is actionable).
- When you identify a problem, try to **suggest a solution** or direction, not just the criticism.
- If a piece of code is clever or well-done, say so — positive feedback is part of a healthy review culture.
- If a change is large and complex, ask the author to walk you through it — a quick sync saves review time.

## Reviewer Checklist

Use this checklist when conducting a review. You do not need to check every item for every PR — use judgment based on what the PR touches. This checklist is for **reviewers**, not authors (see PR templates for the author self-review checklist).

### General

- [ ] PR has a linked Plane Ticket ID and the title clearly describes the change
- [ ] PR description explains *what* changed and *why* — not just *how*
- [ ] PR scope is focused — no unrelated changes bundled in
- [ ] Branch name follows `feature/[Ticket-ID]` or `hotfix/[Ticket-ID]` convention
- [ ] All CI jobs pass (Lint & Format, Unit Test & Coverage, Build)

### Correctness

- [ ] The code does what the PR description claims
- [ ] Edge cases and null/undefined paths are handled
- [ ] Async operations properly handle loading, error, and success states
- [ ] No obvious logic errors or off-by-one bugs

### Security

- [ ] No secrets, tokens, or credentials committed
- [ ] No sensitive data leaked to client bundles
- [ ] Supabase RLS considered for any data access changes
- [ ] User input is validated/sanitised at system boundaries

### Next.js / Architecture

- [ ] `'use client'` is used only where required (interactivity, browser APIs)
- [ ] Data fetching is in Server Components or Server Actions where appropriate
- [ ] Route handlers follow App Router conventions (`app/api/**/route.ts`)
- [ ] No unnecessary client-side bundle bloat introduced

### TypeScript

- [ ] No unjustified `any` types
- [ ] Types are clear, explicit, and consistent with project conventions

### Testing

- [ ] New logic has corresponding unit tests
- [ ] Tests cover meaningful cases (not just the happy path)
- [ ] All tests pass

### Code Quality

- [ ] No dead code, `console.log`, or debug artifacts
- [ ] No over-engineering — solution is proportionate to the problem
- [ ] Names are clear and consistent with existing conventions
- [ ] Documentation/comments updated if behaviour has changed

---

> 💡 **Remember:** The goal of code review is not to find every possible flaw — it's to ship high-quality, safe code as a team. Reviews should make the codebase and the team stronger, not create friction.
